# Fold External Research INTO the Working Doc

## Context

Apr 29-30 commercial reorg org-design benchmarking session. After completing cross-industry research (13 companies, 5 clusters), I built the synthesis as a standalone "external benchmarks layer" companion file that lived alongside the main Apr 29 working doc. When asked the meta-question "should this be folded into the working doc as new sections, or stay separate?", Jon's answer was unambiguous: **"it should all go in one doc."**

The standalone appendix was deleted; content folded into the main working doc as a new Section 13 (External Benchmarks Layer) plus surgical insertions to existing sections (frontmatter, Executive Summary bullet, chart in 5.1, transfer-pricing paragraph in 7.2, six new open questions 9.11-9.16, one new risk 10.11).

## The Learning

For any benchmarking research, external analytical layers, or research companions to in-flight working docs, default to **consolidation INTO the existing doc** rather than standalone companion files.

### Specific structural pattern Jon prefers

When folding external research into a working doc:
1. **Add a new top-level section** for the broader analytical content (e.g., "Section 13: External Benchmarks Layer" with sub-sections 13.1-13.10)
2. **Surgical insertions to existing sections** for specific recommendations (chart updates, paragraph additions, new open questions, new risks)
3. **Renumber existing sections** if needed so the analytical layer fits naturally
4. **Update frontmatter** with new tags + `research_sources` field pointing to underlying research files
5. **Update Executive Summary** to acknowledge the new layer
6. **Delete the standalone companion** once consolidated

### Why this matters

- Working doc reads as ONE artifact when shared with stakeholders (Ross, Ken, Teays)
- Recommendations land where they're load-bearing (transfer pricing in Craig's section; RGM seat in the chart) rather than buried in a companion
- Cross-references stay tight (Section 13.4.1 references Section 9.11)
- Stakeholder doesn't need to read two documents to follow the argument

## When to Apply

- Any benchmarking research delivered alongside an in-flight working doc
- External analytical layers (market mapping, competitive intelligence, M&A target screening) where there's an existing working doc to layer onto
- Any time considering "appendix vs. integrated section" — default to integrated
- Any time considering "companion file vs. embedded" — default to embedded

## Exception

Standalone files are appropriate when:
- The analytical layer is truly independent of any working doc (no host doc exists)
- The analytical work needs to be referenced from multiple working docs over time (then it's a context file or reference, not a deliverable companion)
- The user explicitly asks for a separate deliverable

## Confidence

HIGH. Explicit user statement: "it should all go in one doc." Validated by the resulting consolidation — the working doc now has tighter recommendation→action linkage and reads as one stakeholder-ready artifact.
