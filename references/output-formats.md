# Output Formats

When and how to produce each output type. Don't assume a format — match it to the task or ask Jon.

## When This Applies

Before producing any structured deliverable. Check this file to determine the right format.

## Format Selection

| Situation | Default Format |
|-----------|---------------|
| Exploratory conversation, framing a question | Conversational (no document) |
| Quick competitive scan or screening brief | Conversational with structured sections |
| Strategy analysis Jon will reference later | HTML strategy document |
| Research paper for Jon to review/edit | Word document (.docx) |
| Slide support for a presentation | Concise bullet commentary |
| Data dump Jon will reshape himself | Raw analysis — tables, lists, minimal narrative |

When in doubt, start conversational. Jon will tell you when he wants a document.

## HTML Strategy Documents

Use for substantive analysis Jon will reference or share. Structure:

- Title and date
- Executive summary (3-5 sentences — the "so what" up front)
- Sections with clear headers organized by insight, not by research source
- Key findings highlighted — bold the lead metric or insight in each section
- Assumptions and limitations section at the end
- Clean, professional formatting — no decorative elements

## Word Documents (.docx)

Use for research papers and evaluations Jon may edit, annotate, or forward. Structure:

- Title page with topic, date, and "DRAFT — Internal Use"
- Executive summary
- Body sections with clear hierarchy (H1 for major sections, H2 for subsections)
- Tables for comparative data
- Sources/references section
- Keep under 10 pages unless Jon explicitly asks for more

## Slide Commentary

Use when Jon needs talking points for a presentation. Structure:

- 2-3 bullets per slide maximum
- Lead each bullet with the bolded key metric or insight
- Follow with 1 sentence of context or implication
- No complete sentences as bullets — concise fragments that Jon can talk around
- Flag which slides need data updates

## Raw Analysis

Use when Jon says "just give me the data" or "I'll reshape this." Structure:

- Tables with clear headers
- Minimal narrative — just enough to explain what the data shows
- Source notes for each data point
- No formatting polish — substance over presentation

## Buyer-Facing Materials — Pricing Data Restrictions

When a deliverable will be seen by retail buyers (GroundTruth briefs, category updates, anything for recurring external distribution), GPG's 60-80% US carrot supply share means pricing data becomes negotiating ammunition for the buyer. **Strip the following from any buyer-facing material:**

- Grower price-received trends ($/CWT over time)
- FOB shipping point price tables or YoY percentage changes
- Price differentials between origins (Mexico vs. California)
- Any framing that implies prices rose because of supply concentration
- Production value growth figures that imply price-driven increases (e.g., "$747M to $2B while volume declined")

These are fine for internal analysis (board supplements, Ross briefings, strategy docs). Supply dynamics can be discussed neutrally for buyers — water conditions, seasonal transitions, import patterns, weather — just don't attach price data to them.

When in doubt, ask: "If a buyer read this, would it give them leverage in a pricing negotiation?"

## GroundTruth Buyer Brief — The Three-Version Pattern

Buyer-facing GroundTruth briefs need a balanced tone. Three failure modes to avoid:

1. **Market analyst mode (V1):** Comprehensive data including pricing power signals — gives buyers ammunition (see restrictions above).
2. **Sales pitch mode (V2):** All the reasons to buy more — too one-directional and transparent for recurring distribution. A buyer won't open a monthly email that's obviously a sales pitch.
3. **Trusted update mode (V3 — correct):** "Here's what's happening in your category that you should know." Buyer opens it because it's genuinely useful; GPG benefits because the buyer thinks "this is the supplier who understands my category."

**Structural markers of V3:**
- Full product mix coverage (baby, organic, whole, bulk, value-added — not just GPG's focus segments)
- Weather/growing region conditions section — biggest short-term supply variable
- Consumer spotlight as a rotating feature (one per issue, not a demographic thesis)
- Short-term market dynamics that affect procurement decisions
- "What to Watch" forward-looking section
- No pricing tables, no YoY price increase percentages, no FOB data
- One actionable implication per section (no multi-row recommendation tables)

V3 rules apply to recurring buyer-facing products. Sales decks for specific meetings can be more directional.

## Folding External Research into Working Docs

When external research or benchmarking layers are produced alongside an in-flight working doc, **default to consolidation INTO the existing doc** rather than standalone companion files. Jon's stated preference: "it should all go in one doc."

**Structural pattern:**
1. Add a new top-level section for the broader analytical content (e.g., "Section 13: External Benchmarks Layer" with sub-sections)
2. Surgical insertions to existing sections for specific recommendations (chart updates, paragraph additions, new open questions, new risks)
3. Renumber existing sections if needed so the analytical layer fits naturally
4. Update frontmatter with new tags + `research_sources` field pointing to underlying research files
5. Update Executive Summary to acknowledge the new layer
6. Delete the standalone companion once consolidated

**Why:** Working doc reads as ONE artifact for stakeholders (Ross, Ken, Teays). Recommendations land where they're load-bearing rather than buried in an appendix. Cross-references stay tight.

**Exception — standalone files are appropriate when:**
- The analytical layer is truly independent of any working doc (no host doc exists)
- The analytical work needs to be referenced from multiple working docs over time (then it's a context file or reference, not a deliverable companion)
- The user explicitly asks for a separate deliverable

## Common Pitfalls

- **Format fights.** Producing a 5-page Word doc when Jon wanted a quick conversational take. Default to less formal unless asked.
- **Burying the insight.** In every format, the "so what" comes first. Executive summary, lead bullet, opening sentence — the most important finding is always up top.
- **Over-polishing for a thinking tool.** Jon is the audience, not a board. Clarity matters more than presentation.
- **Companion-file reflex.** Producing a separate analytical deliverable when there's already a working doc to fold it into. Default to integrated.
